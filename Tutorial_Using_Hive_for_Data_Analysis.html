Step 3. Create a Table Using RCFormat

 

The RC (Record-Columnar) File is a column-based file format, to manage the record low in groups. With this format, you can get

the advantages of a columnar format over row format of a record.

The figure below shows the step-by-step picture to help you understand the process of creating the RC file.

 

®Á Create RC file format table.

CREATE TABLE TwitterExampleRCtable(
tweetId INT, username BIGINT,
txt STRING, CreatedAt STRING,
profileLocation STRING COMMENT 'Location of user',
favc INT,retweet STRING,retcount INT,followerscount INT)
COMMENT 'This is the Twitter streaming data'
ROW FORMAT DELIMITED
FIELDS TERMINATED BY '\t'
STORED AS RCFILE;

®Ë Load data into the RC Table. Execute and check the results.

INSERT OVERWRITE TABLE TwitterExampleRCtable select * from TwitterExampletextexample;

Step 4. Query the Tables

Find top ten countries who tweeted most using TwitterExampleRCtable. And check the result.

Select profileLocation, COUNT(txt) as count1 FROM TwitterExampleRCtable GROUP BY profileLocation ORDER BY count1 desc limit 10;

 

 

Step 5. Compare the Managed Tables and External Tables

 

Managed tables are created by default with CREATE TABLE statements, otherwise External tables are used when you want your tables to point to data file in place.

®Á Create the Managed Table.

CREATE TABLE ManagedExample(

        tweetId BIGINT, username STRING,

        txt STRING, CreatedAt STRING,

        profileLocation STRING,

        favc BIGINT,retweet STRING,retcount BIGINT,followerscount BIGINT)

    COMMENT 'This is the Twitter streaming data'

    ROW FORMAT DELIMITED

    FIELDS TERMINATED BY '\t'

    STORED AS TEXTFILE;

 

®Ë Create the External Table.

CREATE EXTERNAL TABLE IF NOT EXISTS ExternalExample(

        tweetId BIGINT, username STRING,

        txt STRING, CreatedAt STRING,

        profileLocation STRING,

        favc BIGINT,retweet STRING,retcount BIGINT,followerscount BIGINT)

    COMMENT 'This is the Twitter streaming data'

    ROW FORMAT DELIMITED

    FIELDS TERMINATED BY '\t'

    STORED AS TEXTFILE

    location '/user/hadoop/TwitterData.txt';

 

When you drop a Managed table, it deletes the data, and it also deletes the metadata.

But External table case, it only deletes the metadata.

®È Compare the output with Managed table and External table, by command for describe.

describe formatted ManagedExample;

describe formatted ExternalExample;

 

 

 

Step 6. Storing as ORCFormat

RC file influenced by the column type, you can apply a SerDe only one record at a time. So each data type can not be applied to the effective compression. In addition, the RC file is inefficient when you look up the specific value of the data type, such as Map, List. This is because because with the RC file is also unnecessary to query the data in Map and List in was designed for sequential record access.

ORC is a file format that improves the shortcomings of these RC files. ORC file has been added from the hive 0.12.0, for certain types of queries show the performance gains over previous file formats. However, a significant amount of time it takes to produce the ORC file compared to the previous file.

.

Create ORC file format table.

CREATE TABLE ORCFileFormatExample(

        tweetId INT, username BIGINT,

        txt STRING, CreatedAt STRING,

        profileLocation STRING COMMENT 'Location of user',

        favc INT,retweet STRING,retcount INT,followerscount INT)

    COMMENT 'This is the Twitter streaming data'

    ROW FORMAT DELIMITED

    FIELDS TERMINATED BY '\t'

    STORED AS ORC tblproperties ("orc.compress"="ZLIB");

 

Step 7. Create a PARTITIONED Table

Partitions are horizontal slices of data which allow large sets of data to be segmented into more manageable blocks.

 

 

®Á Create PARTITIONED table.

CREATE TABLE PARTITIONEDExample(

tweetId INT, username BIGINT, txt STRING,favc INT,retweet STRING,retcount INT,followerscount INT) COMMENT 'This is the Twitter streaming data' PARTITIONED BY(CreatedAt STRING, profileLocation STRING) ROW FORMAT DELIMITED FIELDS TERMINATED BY '\t' STORED AS TEXTFILE;

 

®Ë Load data into PARTITIONED table. Then check the result.

FROM twitterexampletextexample

INSERT OVERWRITE TABLE PARTITIONEDExample PARTITION (CreatedAt="26 04:50:56 UTC 2014",profileLocation="Chicago") SELECT tweetId,username,txt,favc,retweet,retcount,followerscount where profileLocation='Chicago' limit 100;

 

Step 8. Create a Table with Bucket

Hive provides a data model called the bucket to perform an efficient query. Bucketed tables are fantastic in that they allow much more efficient sampling than do non-bucketed tables, and they may later allow for time saving operations such as map side joins. However, the bucketing specified at table creation is not enforced when the table is written to, and so it is possible for the table's metadata to advertise properties which are not upheld by the table's actual layout. This should obviously be avoided.

 

®Á Create the table with Bucket.

CREATE TABLE BucketingExample(

        tweetId INT, username BIGINT,

        txt STRING,CreatedAt STRING,favc INT,retweet STRING,retcount INT,

followerscount INT)

    COMMENT 'This is the Twitter streaming data'

    PARTITIONED BY( profileLocation STRING)

    CLUSTERED BY(tweetId) INTO 2 BUCKETS

    ROW FORMAT DELIMITED

    FIELDS TERMINATED BY '\t'

    STORED AS TEXTFILE;

 

set hive.enforce.bucketing = true;

 

Ref*.) The command °Æset hive.enforce.bucketing = true;°Ø allows the correct number of reducers and the cluster by column to be automatically selected based on the table.

®Ë Load a data to new table. Then check the result.

FROM twitterexampletextexample

    INSERT OVERWRITE TABLE BucketingExample PARTITION (profileLocation="Chicago")   

SELECT tweetId,username,txt,CreatedAt,favc,retweet,retcount,followerscount where profileLocation='Chicago' limit 100;

Conclusion

 

There are two tools for doing SQL-like queries in CDH: Hive and Impala.

Hive works by translating SQL queries into MapReduce jobs, so it's best for large batch jobs and applying flexible transformations.

Impala is significantly faster and is intended to have low enough latency for interactive queries and data exploration.

Impala and Hive can share the metadata about these tables (and their query languages are similar),

so we have used Hive's shell to define these tables, similar to how you would have defined them in your relational database.

Hive's query language will look very familiar if you know SQL, and they are intended to be as compatible as possible.

Then you will run the same query in Impala and compare performance.
